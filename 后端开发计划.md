## 一、技术栈与架构

```
┌─────────────────────────────────────────────┐
│  前端 一家食光.html（现单文件 + Babel CDN）       │
│   · 通过 CDN 引入 @supabase/supabase-js         │
│   · 新增 api 层逐步替换 Mock / sessionStorage    │
└───────────────┬─────────────────────────────┘
                │ HTTPS（anon key + 用户 JWT）
                ▼
┌─────────────────────────────────────────────┐
│  Supabase                                     │
│   · Auth        账号体系（用户名映射邮箱）         │
│   · Postgres    业务数据 + RLS 行级权限           │
│   · Storage     图片（头像/记忆图/胶囊封面）        │
│   · Edge Functions（Deno）  ← AI 代理，藏密钥      │
└───────────────┬─────────────────────────────┘
                │ 服务端调用（密钥仅在 Edge Function）
                ▼
┌─────────────────────────────────────────────┐
│  DeepSeek API   https://api.deepseek.com       │
│   · 对话挖掘 / 故事生成 / 菜谱 / 生平 / 标签        │
└─────────────────────────────────────────────┘
```

### 关键决策

| 项 | 选择 | 理由 |
|----|------|------|
| 账号体系 | **Supabase Auth + 用户名→邮箱映射**（`{username}@yjsg.local`），密保问题存 profiles 表 | 前端用「用户名+密码+密保」，非邮箱；用映射可复用 Auth 的安全密码存储与 JWT，避免自己造轮子 |
| AI 密钥 | **只放在 Edge Function 的 Secrets 里**，前端永远不接触 DeepSeek key | 防止密钥泄露；前端只调用自家 Edge Function |
| 权限 | **RLS 行级安全**为主，敏感操作（创建者专属、AI 调用）在 Edge Function 二次校验 | 即使前端被绕过，数据库层也拦得住 |
| 前端改造 | **不强制先迁 Vite**，先用 CDN 版 supabase-js 接入，逐页替换 Mock | 降低改造成本，保持现有可运行状态 |

## 四、权限（RLS）原则

> 对齐 `需要补充逻辑_整理版.md` §3.7（家族创建者 vs 成员）。

- **基本可见性**：用户只能读写自己所属家族（`family_members` 里有自己 user_id）的数据。
- **创建者专属**（需在 RLS 或 Edge 中校验 `families.creator_id = auth.uid()`）：审核加入/认领申请、改家族信息、重置家族代码、移除成员、转让创建者。
- **本人专属**：普通成员只能改自己的 profile 与「自己创建」的记忆/补述；anchors/comments 的写操作校验 `created_by = auth.uid()`。
- **AI 调用**：对话/故事/菜谱等 Edge Function 内先校验调用者属于该家族，再放行。
- 前端的「隐藏按钮」只是体验，**真正的拦截在 RLS / Edge**。

## 五、AI 集成方案（DeepSeek）

所有 AI 能力都走 **Supabase Edge Function** 代理，密钥在服务端。DeepSeek 是 OpenAI 兼容接口：

```
POST https://api.deepseek.com/chat/completions
Authorization: Bearer $DEEPSEEK_API_KEY
{ "model": $DEEPSEEK_MODEL, "messages": [...], "temperature": ... }
```

| Edge Function | 对应页面 | 任务 | 依据 |
|---------------|---------|------|------|
| `agent-chat` | P30 | 记忆挖掘对话，逐轮回复 + 更新后台状态 | **P30 规格全文**（人设1 / 策略3-7 / 状态8.1 / SystemPrompt 11 / 请求13.2 / 时空校验13.4）|
| `generate-story` | P31 | 把对话整理成结构化故事 | P30 规格 §10.2 输出结构 |
| `generate-recipe` | P11 | 据记忆描述还原菜谱 | 输出 {ingredients[], steps[], note} |
| `generate-bio` | P21 | 生成人物生平简述 | 需要补充逻辑 §AI 场景 |
| `generate-tags` | P11/P31/P_EditMemory | 据正文生成 4-5 个标签 | 需要补充逻辑 §2.5（替换前端 `generateTags`）|

### 5.1 `agent-chat` 实现要点（最关键，单独成步）

```
入参（前端组装，见 P30 §13.2 ConversationRequest）：
  conversation_id, user_message
后端流程：
  1. 取出该对话的 context（family_name/surname、family_tree_nodes 含 hometown、
     related_person_hometown、existing_anchors、gray_anchors、entry_type/context）
  2. 取出 conversation_messages 历史 + 当前 state（§8.1 JSON）
  3. 按 §11 System Prompt 模板填充变量，组装 messages
  4. 调 DeepSeek → 得到自然语言回复（绝不让模型输出 JSON 给用户）
  5. 用第二次轻量调用 / 规则，从本轮抽取 era / city 等，更新 state（§13.4 校验：
     时间未锚定→下轮温和追问；地点未到市级→用籍贯锚点追问）
  6. 落库：插入 user + assistant 两条 message，更新 conversations.state
  7. 返回：assistant 文本 + 是否给出「整理成故事」CTA（状态足够丰富时）
```

> 红线（来自规格 §1.3 / §5）：禁止填表口吻问时间地点；禁止主动问生死；一次只问一个核心问题；不自称 AI；不输出结构化内容给用户。

### 5.2 `generate-story` 实现要点

```
入参：conversation_id + 关联归属（送给谁/署名）
流程：把 history + state 喂给 DeepSeek，要求产出 §10.2 的 JSON（food/people/
      scene.location(含 city、geo_point)/emotion/related_foods/story_suggestions/
      gray_anchor_suggestions）→ 存 stories 表
      → 用户点「点亮」时：据此创建 anchors 记录（status=lit），
        并把 gray_anchor_suggestions 落为新的灰锚点（status=gray）
```

## 七、开发纪律

1. **一次只做一步**：完成 → 自测 → 在 `后端开发记录.md` 记一条（做了什么、表/函数名、验收结果）→ 再开下一步。
2. **每步先建表+RLS，再写读写逻辑，最后接前端**，避免一次性大改。
3. **AI 步骤（4.3/4.4）务必对照 P30 规格逐条核对人设与红线**，宁可慢，不可走样。
4. **密钥安全**：DeepSeek key 只在 Edge Secrets；service key 不进前端。
5. **不破坏现有可运行前端**：接入一页验证一页，Mock 与真实数据可短期并存。
