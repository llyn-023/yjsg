## 1. 概述与设计原则

### 1.1 核心设计理念

**用户通过 6 种基础关系「钉」自己在家谱中的位置，所有亲属称谓由系统自动推导。**

用户在建立关系时，只能从 6 种中选择：父亲、母亲、儿子、女儿、丈夫、妻子。这 6 种关系唯一的作用是**定位**——把自己"挂"到家谱树上某个已有节点的旁边。

挂上去之后，系统从这棵树的结构自动推导出：
- 家谱上所有人分别是我的什么（长辈、同辈、晚辈）
- 我分别是家谱上所有人的什么（反向关系）
- 任意两个人之间的称谓（用于通知、动态等）

> 用户做的事：选一个节点 + 选一种关系 → 定位。
> 系统做的事：从整棵树的结构推导出所有人两两之间的称谓。

### 1.2 三个核心问题

| 问题 | 说明 |
|------|------|
| **存储** | 只存「父子」和「配偶」两种基础关系边 |
| **推导** | 从基础关系链推导出任意两人之间的亲属称谓（双向） |
| **应用** | 建档定位、家谱图渲染、通知提醒直接使用推导结果 |

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| **6 种关系仅用于定位** | 用户不声明"爷爷""叔叔"等，这些全部由系统推导 |
| **双向对称推导** | 推导 A→B 的同时，必须推导 B→A（叔叔↔侄子、堂兄↔堂弟） |
| **性别决定称谓** | 父亲 vs 母亲、爷爷 vs 外公，完全由路径上每个人的性别决定 |
| **堂表由血脉路径决定** | 堂 vs 表 不是看共同祖先是谁，而是看连接路径是否纯父系（详见第 6 节） |
| **长幼由出生日期决定** | 同辈哥/弟、姐/妹、伯/叔，统一用完整出生年月日比较（详见第 7 节） |
| **统一标准** | 全系统使用同一套标准称谓，不支持自定义 |
| **缓存优先** | 亲属称谓推导后缓存到表中，通知等场景直接查表 |
| **纠正级联** | 关系纠正后自动重推导所有受影响的称谓 |

## 2. 核心数据模型

### 2.1 家族成员节点表（family_member）

```sql
CREATE TABLE family_member (
    id            VARCHAR(36) PRIMARY KEY,
    family_id     VARCHAR(36) NOT NULL,
    name          VARCHAR(50) NOT NULL,
    gender        ENUM('male', 'female') NOT NULL,
    birth_date    DATE,                            -- 出生年月日（可选，用于长幼排序）
    birth_type    ENUM('solar', 'lunar') DEFAULT 'solar',  -- 阳历/农历
    death_date    DATE,                            -- 忌日（null = 在世）
    status        ENUM('active', 'placeholder', 'deceased') NOT NULL,
    user_id       VARCHAR(36),                    -- 关联账号（null = 未认领）
    avatar_url    VARCHAR(500),
    created_at    DATETIME NOT NULL,
    updated_at    DATETIME NOT NULL,

    INDEX idx_family (family_id),
    INDEX idx_user (user_id)
);
```

**字段说明**：

| 字段 | 说明 |
|------|------|
| `gender` | 必填。创建时由用户选择或由定位关系推导得出。**堂表判定与长幼排序的核心输入** |
| `birth_date` | 完整出生年月日，用于同辈长幼排序（哥/弟、姐/妹、伯/叔）。**无此字段时降级排序**（见第 7 节） |
| `birth_type` | 阳历/农历。农历需统一换算为阳历后再比较长幼 |
| `status` | `active`=已认领、`placeholder`=占位节点、`deceased`=已故 |

### 2.2 基础关系边表（family_relation）

只存两种边：`parent_of` 和 `spouse_of`。

```sql
CREATE TABLE family_relation (
    id            VARCHAR(36) PRIMARY KEY,
    family_id     VARCHAR(36) NOT NULL,
    relation_type ENUM('parent_of', 'spouse_of') NOT NULL,
    from_node_id  VARCHAR(36) NOT NULL,           -- parent_of: 父/母；spouse_of: 任意一方
    to_node_id    VARCHAR(36) NOT NULL,           -- parent_of: 子女；spouse_of: 另一方
    created_at    DATETIME NOT NULL,

    UNIQUE KEY uk_relation (family_id, relation_type, from_node_id, to_node_id),
    INDEX idx_from (from_node_id),
    INDEX idx_to (to_node_id),
    INDEX idx_family (family_id)
);
```

| 类型 | 含义 | 方向 |
|------|------|------|
| `parent_of(A, B)` | A 是 B 的父/母 | 有向：A→B |
| `spouse_of(A, B)` | A 和 B 是配偶 | 无向（只存一条） |

### 2.3 亲属称谓缓存表（kinship_cache）

预计算的双向称谓结果。关系变更时批量更新。

```sql
CREATE TABLE kinship_cache (
    id            VARCHAR(36) PRIMARY KEY,
    family_id     VARCHAR(36) NOT NULL,
    ego_id        VARCHAR(36) NOT NULL,           -- "谁在看"
    alter_id      VARCHAR(36) NOT NULL,           -- "看谁"
    kinship_term  VARCHAR(20) NOT NULL,           -- 称谓（如"外婆"）
    updated_at    DATETIME NOT NULL,

    UNIQUE KEY uk_pair (family_id, ego_id, alter_id),
    INDEX idx_ego (family_id, ego_id),
    INDEX idx_alter (family_id, alter_id)
);
```

**关键**：对于任意两个人 A 和 B，缓存中有**两行**：

```
ego=A, alter=B, kinship_term="叔叔"    （A 看 B 是叔叔）
ego=B, alter=A, kinship_term="侄子"    （B 看 A 是侄子）
```

## 3. 基础关系定义

### 3.1 为什么只有两种基础关系？

| 基础关系 | 说明 | 为什么足够 |
|---------|------|-----------|
| `parent_of(A, B)` | A 是 B 的父/母 | 所有血亲关系通过父子链推导 |
| `spouse_of(A, B)` | A 和 B 是配偶 | 区分父系/母系，推导姻亲 |

**兄弟** → 共享父母推导。
**叔伯姑舅** → 父母辈的父子链推导。
**堂表** → 祖辈的父子链 + **路径性别**推导（详见第 6 节）。

### 3.2 性别如何决定称谓

```
parent_of(A, B):
  A 是男性 → A 是 B 的「父亲」
  A 是女性 → A 是 B 的「母亲」

spouse_of(A, B):
  B 是男性 → B 是 A 的「丈夫」
  B 是女性 → B 是 A 的「妻子」
```

**父系 vs 母系**：

```
X 是 ego 的「爷爷」还是「外公」？
  → 路径经过 ego 的父亲 → 爷爷
  → 路径经过 ego 的母亲 → 外公
```

### 3.3 性别字段是堂表判定的关键

> ⚠️ `parent_of(A, B)` 中 **A 的性别**（男性=父亲 / 女性=母亲）不仅决定直系称谓，更是第 6 节堂表判定的核心输入。系统必须在创建边时确认或推导每个节点的性别。

## 4. 定位声明映射

### 4.1 用户可选的 6 种关系

用户在 P3 加入家族或 P20 添加家人时，通过以下流程**定位**：

```
第一步：选择自己的性别（男 / 女）
第二步：选择一位已有成员（如"陈小云"）
第三步：从 6 个选项中选择定位关系
        ┌─────────────────────────────┐
        │  父亲   母亲   儿子   女儿  │
        │       丈夫       妻子       │
        └─────────────────────────────┘
第四步：确认关系
```

> ⚠️ 这 6 种关系的唯一作用是把用户「钉」到家谱树上。钉好之后，所有亲属称谓由系统推导。

### 4.2 六种定位关系的基础边转换

| 用户选择 | 含义 | 创建的基础边 | 确认的性别信息 |
|---------|------|------------|--------------|
| **父亲** | X 是我的父亲 | `parent_of(X, ego)` | X.gender = male |
| **母亲** | X 是我的母亲 | `parent_of(X, ego)` | X.gender = female |
| **儿子** | X 是我的儿子 | `parent_of(ego, X)` | X.gender = male |
| **女儿** | X 是我的女儿 | `parent_of(ego, X)` | X.gender = female |
| **丈夫** | X 是我的丈夫 | `spouse_of(ego, X)` | X.gender = male |
| **妻子** | X 是我的妻子 | `spouse_of(ego, X)` | X.gender = female |

**用户性别的作用**：
1. 创建 ego 节点的 gender 字段
2. 作为推导链条中的一环（用户是女性 → 用户的儿子有外孙，用户的女儿有外孙女）
3. 性别校验（男性不能选"丈夫"，女性不能选"妻子"）

### 4.3 一次定位引发的连锁推导示例

```
已有家谱：
  爷爷(陈大明,M) ─── 奶奶(王秀兰,F)
       │
  父亲(陈建国,M) ─── 母亲(李芳,F)
       │
  哥哥(陈小云,M)

新用户（女性）加入，选择「陈小云 是我的——父亲」：

创建边：parent_of(陈小云, ego)
确认：陈小云.gender = male，ego.gender = female

系统自动推导（ego 视角）：
  陈建国 → ego 的爷爷
  王秀兰 → ego 的奶奶
  李芳   → ego 的母亲（陈小云的配偶）
  陈大明 → ego 的曾祖父
  王秀兰(祖辈) → ego 的曾祖母

系统自动推导（反向——别人看 ego）：
  ego → 陈小云的女儿
  ego → 陈建国的孙女
  ego → 李芳的女儿
  ego → 陈大明的曾孙女

双向 kinship_cache 全部写入。
```

### 4.4 不存在中间节点占位问题

因为用户只能通过 6 种一阶关系定位，**永远不会出现需要补中间节点的情况**。

如果某个中间节点（如父亲）不在谱中：
- 用户先找另一个在谱中的节点定位（如哥哥 → 选"兄弟"）
- 或者等家族其他人把中间节点添加进来

## 5. 亲属称谓推导算法

### 5.1 算法概述

```
输入：ego_id, alter_id
输出：kinship_term（如"外婆"）

步骤：
1. 在关系图中找到 ego → alter 的最短路径
2. 将路径翻译为方向序列（↑上、↓下、↔旁）
3. 根据方向序列 + 路径上每个人的性别，查称谓映射表
4. 对于同辈旁系（↑步数=↓步数），额外判定堂/表（第 6 节）
5. 返回称谓
```

**核心要求**：对任意两个人 A 和 B，必须计算两次：
- `derive(A, B)` → A 看 B 是什么（如"叔叔"）
- `derive(B, A)` → B 看 A 是什么（如"侄子"）

### 5.2 关系图构建

```typescript
interface RelationGraph {
    nodes: Map<string, {
        gender: 'male' | 'female';
        birthDate?: Date;
        birthType?: 'solar' | 'lunar';
    }>;
    edges: Map<string, Array<{
        target: string;
        type: 'parent_of' | 'spouse_of';
        direction: 'up' | 'down' | 'side';
    }>>;
}
```

**边的方向**：

| 基础边 | 从子节点出发 | 从父节点出发 |
|--------|------------|------------|
| `parent_of(A, B)` | B→A, direction='up' | A→B, direction='down' |
| `spouse_of(A, B)` | 双向, direction='side' | 双向, direction='side' |

### 5.3 最短路径查找

使用 **BFS** 查找最短路径。多条等长路径时，**优先选择不含配偶边的路径**（血亲优先于姻亲）。

```typescript
interface PathStep {
    direction: 'up' | 'down' | 'side';
    nodeId: string;
    gender: 'male' | 'female';
    birthDate?: Date;
    // 关键：记录这条边的"父节点性别"（用于堂表判定）
    parentGender?: 'male' | 'female';  // 仅 direction='up' 时有效
}

function findShortestPath(
    graph: RelationGraph,
    egoId: string,
    alterId: string
): PathStep[] | null;
```

### 5.4 路径翻译为称谓

```typescript
function translatePath(path: PathStep[], egoGender: string): string {
    if (path.length === 0) return '自己';
    const directions = path.map(step => step.direction);

    // 同辈旁系需要堂表判定
    if (isCousinPattern(directions)) {
        return deriveCousinTerm(path, egoGender);  // 第 6 节
    }

    return lookupKinshipTerm(directions, path, egoGender);
}
```

### 5.5 递归推导规则（任意深度）

**规则一：直系长辈（纯 ↑）**

| 步数 | 前缀 | 性别链决定直系/外系 |
|------|------|-------------------|
| 1 ↑ | — | 父亲 / 母亲 |
| 2 ↑ | — | 爷爷/奶奶/外公/外婆 |
| 3 ↑ | 曾 | 曾祖父/曾祖母/外曾… |
| 4 ↑ | 高 | 高祖父/高祖母/外高… |
| 5 ↑+ | 天/烈/太/远/鼻 | 按性别组合确定 |

> 超过 5 代统一返回"长辈"。

**规则二：直系晚辈（纯 ↓）**

| 步数 | 前缀 | 第一节决定直系/外系 |
|------|------|-------------------|
| 1 ↓ | — | 儿子 / 女儿 |
| 2 ↓ | — | 孙子/孙女/外孙/外孙女 |
| 3 ↓ | 曾 | 曾孙/曾孙女/外曾孙… |
| n ↓ | 同长辈前缀 | 儿子→直系，女儿→外系 |

**规则三：旁系同辈（↑ 步数 = ↓ 步数）** → **见第 6 节堂表判定**

**规则四：旁系长辈（↑ 比 ↓ 多 1 步）**

| 经由 | 目标性别 | 年龄关系 | 称谓 |
|------|---------|---------|------|
| 父亲 | 男 | 年长 | 伯父 |
| 父亲 | 男 | 年幼 | 叔叔 |
| 父亲 | 女 | — | 姑姑 |
| 母亲 | 男 | — | 舅舅 |
| 母亲 | 女 | — | 阿姨（姨妈） |

**规则五：旁系晚辈（↓ 比 ↑ 多 1 步）**

| 经由 | 目标性别 | 称谓 |
|------|---------|------|
| 兄弟 | 男 | 侄子 |
| 兄弟 | 女 | 侄女 |
| 姐妹 | 男 | 外甥 |
| 姐妹 | 女 | 外甥女 |

**规则六：姻亲（路径含 ↔）**

| 路径 | ego 性别 | 称谓 |
|------|---------|------|
| ↔↑ | 女 | 公公 / 婆婆 |
| ↔↑ | 男 | 岳父 / 岳母 |
| ↔↑↓ | 女 | 大伯子/小叔子/大姑子/小姑子 |
| ↔↑↓ | 男 | 大舅子/小舅子/大姨子/小姨子 |
| ↑↓↔ | — | 嫂子/弟妹/姐夫/妹夫 |
| ↓↔ | — | 儿媳 / 女婿 |

## 6. 堂/表判定规则（核心）

> 本节解决最关键的问题：同辈旁系亲属（共同祖辈）何时称"堂"，何时称"表"。

### 6.1 一句话原则

> **堂亲**：我和 TA 的父亲是亲兄弟（同祖父、同姓、纯父系血脉相连）。
> **表亲**：其余所有同辈旁系（姑表 / 舅表 / 姨表）。

### 6.2 根本区别：血脉路径是否"纯父系"

堂与表的区别**不是**"共同祖先是祖父还是外祖父"，而是**连接我和 TA 的血脉路径是否全程经过男性（父→子）**。

设共同祖先为 A（我和 TA 的最近共同祖先，通常是祖辈）：
- 从 A 到我的路径：`A → ... → 我`
- 从 A 到 TA 的路径：`A → ... → TA`

判定：

| 两条路径情况 | 结论 |
|------------|------|
| 两条都**全程父→子**（每一步的父母都是男性父亲） | **堂亲** |
| 任一条**中途经过母亲/姑姑/姨妈**（女性血脉转折） | **表亲** |

### 6.3 四种同辈旁系的精确判定

#### ① 堂兄弟姐妹（父系·同姓）

```
共同祖先 = 祖父（父亲的父亲，男性）
路径全程男性血脉：

  祖父 ──┬── 父亲(男) ── 我
         │
         └── 叔叔/伯父(男) ── 堂兄弟姐妹

判定要点：
  - 我的父亲 和 TA 的父亲 是 兄弟关系
  - 祖父→父亲：父→子 ✓
  - 祖父→叔伯：父→子 ✓
  - 全程父系 → 堂
```

#### ② 姑表兄弟姐妹（父亲的姐妹的子女）

```
共同祖先 = 祖父
但路径经过姑姑（女性）：

  祖父 ──┬── 父亲(男) ── 我
         │
         └── 姑姑(女) ──[嫁出]── 姑表兄弟姐妹

判定要点：
  - 祖父→姑姑：父→女 ✗（经过女性）
  - → 表（姑表）
```

#### ③ 舅表兄弟姐妹（母亲的兄弟的子女）

```
共同祖先 = 外祖父
路径经过母亲（女性）：

  外祖父 ──┬── 母亲(女) ── 我
           │
           └── 舅舅(男) ── 舅表兄弟姐妹

判定要点：
  - 外祖父→母亲：父→女 ✗（经过女性）
  - → 表（舅表）
```

#### ④ 姨表兄弟姐妹（母亲的姐妹的子女）

```
共同祖先 = 外祖父/外祖母
路径经过母亲和姨妈（女性）：

  外祖父 ──┬── 母亲(女) ── 我
           │
           └── 姨妈(女) ── 姨表兄弟姐妹

判定要点：
  - 外祖父→母亲：父→女 ✗
  - → 表（姨表）
```

### 6.4 算法实现：堂表判定函数

```typescript
/**
 * 判定 ego 和 peer（同辈旁系）是堂亲还是表亲
 * @returns 'tang' | 'biao'
 */
function classifyCousin(
    graph: RelationGraph,
    egoId: string,
    peerId: string
): 'tang' | 'biao' {
    // 1. 找最近共同祖先（LCA）
    const lca = findLowestCommonAncestor(graph, egoId, peerId);
    if (!lca) return 'biao';  // 无共同祖先，按表亲兜底

    // 2. 取 LCA → ego 和 LCA → peer 两条路径
    const pathToEgo = pathFromAncestor(graph, lca, egoId);
    const pathToPeer = pathFromAncestor(graph, lca, peerId);

    // 3. 检查两条路径是否"全程父→子"
    //    即路径上每一节的 parent_of 边，from 节点都是男性（父亲）
    const egoAllPatriarch = pathToEgo.every(step =>
        step.direction === 'down' && step.parentNode?.gender === 'male'
    );
    const peerAllPatriarch = pathToPeer.every(step =>
        step.direction === 'down' && step.parentNode?.gender === 'male'
    );

    // 4. 两条都纯父系 → 堂；否则 → 表
    return (egoAllPatriarch && peerAllPatriarch) ? 'tang' : 'biao';
}
```

### 6.8 常见误区纠正

| 误区 | 正解 |
|------|------|
| "共同祖先是祖父就是堂" | ❌ 若经由姑姑（父亲的姐妹）连接，仍是表（姑表） |
| "共同祖先是外祖父就是表" | ✅ 但本质原因是路径经过母亲（女性），不是"外祖父"这个身份 |
| "同姓就是堂" | 大致正确但不严谨。本质是"父亲们是兄弟"，姓氏是结果不是判定依据 |
| "大伯的孩子是表亲" | ❌ 大伯（父亲的兄弟）的孩子是**堂**亲。原型中此类标注需纠正 |

## 7. 长幼排序规则

### 7.1 核心原则

> 同辈之间（兄弟姐妹、堂表兄弟姐妹、伯叔等），**谁更年长谁为"哥/姐/伯"**，由**完整出生年月日**确定。

### 7.2 排序依据

| 数据情况 | 比较方式 |
|---------|---------|
| 双方都有 `birth_date` | 直接比较日期，**日期小的（先出生的）年长** |
| 一方有 `birth_date`，另一方无 | **有日期的视为年长**（信息明确者优先） |
| 双方都只有 `birth_year` | 比较年份，年份小的年长 |
| 双方都无任何出生信息 | 按**加入家谱的先后顺序**（`created_at`），先加入的年长 |
| 农历 `birth_type='lunar'` | **先换算为阳历**再比较 |

### 7.3 影响的称谓

| 关系 | 我年长于TA | TA年长于我 |
|------|-----------|-----------|
| 亲兄弟 | TA 是我弟弟 | TA 是我哥哥 |
| 亲姐妹 | TA 是我妹妹 | TA 是我姐姐 |
| 堂兄弟 | TA 是我堂弟 | TA 是我堂兄 |
| 堂姐妹 | TA 是我堂妹 | TA 是我堂姐 |
| 表兄弟 | TA 是我表弟 | TA 是我表兄 |
| 表姐妹 | TA 是我表妹 | TA 是我表姐 |
| 父亲的兄弟 | TA 是我叔叔（年幼于父亲） | TA 是我伯父（年长于父亲） |

### 7.4 排序算法

```typescript
/**
 * 比较 ego 和 peer 的长幼
 * @returns 'ego_older' | 'peer_older' | 'unknown'
 */
function compareAge(
    ego: FamilyMember,
    peer: FamilyMember
): 'ego_older' | 'peer_older' | 'unknown' {
    const egoDate = normalizeBirthDate(ego);  // 农历转阳历
    const peerDate = normalizeBirthDate(peer);

    // 双方都有完整日期
    if (egoDate && peerDate) {
        return egoDate < peerDate ? 'ego_older' : 'peer_older';
    }
    // 一方有，另一方无 → 有的一方年长
    if (egoDate && !peerDate) return 'ego_older';
    if (!egoDate && peerDate) return 'peer_older';
    // 双方都无 → 按加入顺序
    if (ego.created_at < peer.created_at) return 'ego_older';
    if (ego.created_at > peer.created_at) return 'peer_older';
    return 'unknown';
}

// 农历换算为阳历（需引入农历库）
function normalizeBirthDate(m: FamilyMember): Date | null {
    if (!m.birth_date) return null;
    if (m.birth_type === 'lunar') {
        return lunarToSolar(m.birth_date);  // 农历转阳历
    }
    return new Date(m.birth_date);
}
```

### 7.5 伯/叔的特殊性

> "伯父"vs"叔叔"是**父亲的兄弟相对于父亲**的长幼，不是相对于我。

```
父亲 有一个哥哥（年长于父亲）→ 这个哥哥是我的「伯父」
父亲 有一个弟弟（年幼于父亲）→ 这个弟弟是我的「叔叔」
```

判定时：比较 **TA 与我父亲** 的出生日期，而非 TA 与我。

## 9. 双向推导与反向称谓表

### 9.1 双向推导原则

对于家族中任意两个人 A 和 B，`kinship_cache` 中必须写入**两行**：

```
ego=A, alter=B → A 看 B 是什么称谓
ego=B, alter=A → B 看 A 是什么称谓
```

这两个称谓通常**不同**：

```
A 看 B → "叔叔"
B 看 A → "侄子"

A 看 B → "堂兄"
B 看 A → "堂弟"

A 看 B → "奶奶"
B 看 A → "孙子"
```

### 9.3 双向推导的完整示例

```
家谱结构：
  爷爷(M) ─── 奶奶(F)
       │
  叔叔(M)   父亲(M) ─── 母亲(F)
       │         │
  堂兄(M)    哥哥(M)   我(F)

ego = 我（女性）

推导结果（kinship_cache 中 ego=我 的所有行）：
  我 → 父亲    = 父亲
  我 → 母亲    = 母亲
  我 → 哥哥    = 哥哥
  我 → 爷爷    = 爷爷
  我 → 奶奶    = 奶奶
  我 → 叔叔    = 叔叔
  我 → 堂兄    = 堂兄        ← 第6节判定：父亲们是兄弟 → 堂

反向推导（kinship_cache 中 ego=其他人, alter=我）：
  父亲 → 我    = 女儿
  母亲 → 我    = 女儿
  哥哥 → 我    = 妹妹
  爷爷 → 我    = 孙女
  奶奶 → 我    = 孙女
  叔叔 → 我    = 侄女
  堂兄 → 我    = 堂妹
```

### 9.4 算法实现要点

反向称谓不需要单独的推导算法。只需调用 `derive(B, A)` 即可得到 B 看 A 的称谓。算法本身是无方向的——给它两个节点，它自动根据路径和性别计算结果。

```typescript
// 批量推导：为整个家族生成所有双向称谓
async function buildKinshipCache(familyId: string) {
    const members = await getAllMembers(familyId);

    for (const ego of members) {
        for (const alter of members) {
            if (ego.id === alter.id) continue;

            // 正向
            const term = await deriveKinship(familyId, ego.id, alter.id);
            await upsertCache(familyId, ego.id, alter.id, term);
        }
    }
    // 每个 (A, B) 对被计算两次：derive(A,B) 和 derive(B,A)
    // 结果自然不同，因为路径方向和性别链不同
}
```

async function getKinshipTerm(

### 14.4 堂表判定（核心，详见第 6 节）

```typescript
/**
 * 同辈旁系堂表判定
 * 堂：两条血脉路径都全程父→子（父亲们是兄弟）
 * 表：至少一条经过女性（姑/母/姨）
 */
function classifyCousin(graph, egoId, peerId): CousinClass {
    const lca = findLCA(graph, egoId, peerId);
    if (!lca) return 'biao';

    const toEgo = pathFromAncestor(graph, lca, egoId);
    const toPeer = pathFromAncestor(graph, lca, peerId);

    const allPatriarch = (path) => path.every(step =>
        step.direction === 'down' && step.parentNode.gender === 'male'
    );

    return (allPatriarch(toEgo) && allPatriarch(toPeer)) ? 'tang' : 'biao';
}
```
