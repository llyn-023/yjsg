## 一、现状评估

### 1.1 原有文件

| 文件 | 行数 | 用途 |
|------|------|------|
| `一家食光.html` | ~3896 行 | 完整原型：32 个 React 组件 + 设计令牌 + 内联样式 + Mock 数据，依赖 Babel Standalone 在浏览器端编译 JSX |
| `app-root.jsx` | ~196 行 | 设计画布画廊框架，仅含 P0-P3 四个屏幕 |
| `screen-p0~p3.jsx` | 各 ~200 行 | 4 个画廊展示用屏幕文件 |
| `shared.jsx` | ~212 行 | 画廊版共享组件（C、StatusBar、BottomSheet、Toast、Btn、AppInput、StepDots、NavBar） |

### 1.2 问题诊断

1. **单文件巨石**：3896 行 HTML 混合了样式、32 个组件、Mock 数据、路由逻辑，无法分工协作
2. **无模块化**：依赖 window 全局变量（`window.P10_Home = ...`）通信，组件间耦合混乱
3. **无状态管理**：数据流分散在 sessionStorage、window 临时变量、组件 local state 之间
4. **Mock 数据硬编码**：散落各处，无统一数据层
5. **缺少规格实现**：三份规格文档（家谱关系系统、Agent 对话 Skill、功能逻辑）中的核心算法均未在前端落地
6. **无构建工具**：依赖浏览器端 Babel 编译，无法使用 TypeScript、代码分割、Tree-shaking

## 三、工程化目标架构

### 3.1 技术选型

| 层面 | 选择 | 理由 |
|------|------|------|
| 框架 | React 18 | 与原型一致，生产可用 |
| 语言 | TypeScript (渐进) | 核心数据模型、算法用 TS，组件先用 JSX |
| 状态管理 | React Context + useReducer | 无需引入 Redux，满足中型应用 |
| 样式方案 | CSS-in-JS (内联 → 提取) | 先保留内联风格，逐步提取为 CSS Modules |
| 构建工具 | Vite | 快速开发，支持 JSX/TSX/热更新 |
| Mock 数据 | 统一 mock/ 目录 | 结构化导出，替换散落硬编码 |
| 路由 | 自研 hash 路由 (保持) | 原型已有 workable 方案，增强即可 |

### 3.3 数据流架构

```
用户操作
  → Page Component (useState / useReducer)
    → Context (FamilyContext / AuthContext)
      → engine/ 算法模块 (kinship-derive, tang-biao)
        → Mock Data (开发) / API (生产)
      ← 计算结果回写 Context
    ← 组件重渲染
  ← UI 更新
```

## 十二、去掉设计画廊，改为真实可用前端（V1.4 · 2026-06-13）

### 12.1 背景

原 `一家食光.html` 的根组件是 `Gallery`（设计画廊外壳）：左侧页面目录栏 + 中央等比缩放舞台 + `←/→` 键盘翻页 + 手机边框，用于设计评审逐屏预览。真正的应用主体是被嵌入舞台、固定 `390×844` 的 `PhoneApp`。本轮把运行入口从「画廊」切换为「真实 App」。

### 12.2 改动清单

| 改动 | 详情 |
|------|------|
| **删除 `Gallery`** | 整段移除：左侧目录栏、缩放舞台、键盘翻页、`ResizeObserver` 等比缩放、手机边框/阴影。约 115 行设计画廊专用代码 |
| **新增 `App` 真实外壳** | 渲染入口 `root.render(<App/>)`：响应式——**窄屏（≤520px）铺满整个视口**；**宽屏（>520px）居中为最大 430px 宽的手机列**，监听 `resize` 自适应切换 |
| **`PhoneApp` 容器** | 由固定 `width:PW, height:PH`（390×844）改为 `width/height:100%`，填满父容器 |
| **清理死代码** | 删除已无用的 `const PW=390, PH=844` 及过时注释 |
| **始终从启动页进入** | `PhoneApp` 初始页固定 `P0`，并在挂载时 `history.replaceState` 清除 URL 中残留的旧 hash —— 修复「刷新后停留在旧页面、跳过 P0 启动页」的问题 |
| **网页端两侧配色** | 宽屏两侧留白由深色 `#1A1A1A` 改为浅绿 `C.primaryPale`（`#EBF5F0`），手机列投影改柔和墨绿 `rgba(40,91,78,0.18)`，与主题统一 |

### 12.3 行为说明

- **路由**：仍为自研 hash 路由，应用内前进/后退正常更新 hash；但每次**整页加载**都重置到 P0（无真实会话，启动→登录是预期入口流程）。
- **使用方式**：浏览器直接打开 `一家食光.html`（React/Babel 走 CDN，需联网）。窄窗口全屏、宽窗口居中手机列。
- 全部页面、Mock 数据、交互逻辑保持不变，仅去除画廊外壳。
