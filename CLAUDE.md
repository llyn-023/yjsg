# 一家食光后端 · Claude Code 工作纪律（每个任务都遵守）

## 每次开工前（不可跳过，但要快）
1. 先扫这 4 个文件：后端开发计划.md、后端开发记录.md、前端开发记录.md §十一、需要补充逻辑_整理版.md。
2. 按当前 Phase 追加读：Phase2→家谱关系系统规格全文；Phase4→P30 规格全文；其余 Phase 只扫对应小节。
3. 动手前用一句话说明：现在做哪一步、完成判据是什么。然后直接开干，不必逐句复述文件。

## 开发方式（目标：快而不塌）
- 一个 /goal 内的多个子步骤连续做完，不要每个子步骤都停下等我。每步：建表+RLS → 读写逻辑 → 接前端 → 自测。
- 低风险 Phase（3 / 5 / 7）每个 Phase 结束记一次 后端开发记录.md 即可（做了什么、表名·函数名、验收证据）。
  关键步骤（2.5、4.3、6.1）必须单独详细记录。
- 不实现 需要补充逻辑_整理版.md §八「后端不做」清单里的功能；不加计划外功能。
- 不破坏现有可运行前端，逐页替换 Mock，可短期并存。
- 显示「关系·名字」的地方一律读 kinship_cache，不重新推导。

## 快速验证（核心：又快又真，禁止假装通过）
- 维护一个可复用的冒烟脚本 `scripts/smoke.mjs`（必要时配 .sql）：用 service_role key 一次性建好测试数据、
  跑一组断言、打印每条 PASS/FAIL。每个 Phase 扩充它；之后每步重跑它来快速验证，而不是每次手点。
- 固定两组测试身份：**userA = 家族 F1 的创建者；userB = 家族 F2 的创建者**。
  **越权测试 = 用 userB 的 JWT 去读/写 F1 的数据，必须全部被拒。** 这是每个写接口的硬性检查。
- 真实账号：测试用户用 service_role 的 admin createUser 或 anon 的 signUp 真实创建，不许 mock JWT。
- “通过”必须有真实证据：贴出实际命令 + 真实输出（HTTP 状态码、SQL 返回的真实行、Edge Function 真实响应）。
  只说“应该没问题 / 逻辑已写好”= 未通过。
- 禁止为让测试变绿而：mock 掉被测对象、放宽断言、删测试、跳过 RLS。
- 能脚本化就脚本化、能批量断言就批量，不做慢的交互式探索。把验证时间压到最短，
  但**不许省掉这两项：越权测试、打真实环境**。

## 测试数据卫生（这是真实项目，别污染）
- 所有测试数据加前缀：用户名 `test_*`、家族名 `TEST_*`。每个 Phase 末用脚本清掉本 Phase 造的测试数据。
- 绝不在真实项目上跑 `db reset` / `drop` / 删真实数据 / force push —— 需要时停下问我。

## 安全红线（每步自查）
- GLM_API_KEY / service_role key 绝不进前端、HTML、git。每个 Phase 末 `grep -ri` 全仓确认一次（贴结果）。
- 每张表都有 RLS 并被真实越权测试拦过。敏感操作（创建者专属、AI 调用）在 Edge 二次校验。

## 遇到这些，停下问我，不要编造或硬绕
- 缺凭据/密钥，或某步必须我手动在控制台操作。
- 需要破坏性操作。
- 计划/规格有歧义或冲突。
- 同一测试反复修 3 次仍不过 —— 报告卡点，绝不靠改弱测试糊弄。
- **2.5（kinship_cache）和 4.3（agent-chat）做完后先停，等我亲自 review 再继续；其余 Phase 记录好即可自行继续下一步。**
